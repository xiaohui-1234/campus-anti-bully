# 校园防霸凌系统开发者接口与字段说明书

最后更新：2026-05-25

本文面向接手维护、联调和二次开发人员，说明校园防霸凌系统的接口、字段用途、协议约束、状态流转和常见扩展点。项目包含 Spring Boot 后端、微信小程序、Vue 管理后台、MySQL、Redis、MinIO 和 MQTT。

## 1. 关键原则

开发和二次开发必须遵守以下约束：

1. JSON 对外字段使用 `snake_case`。Java 内部可以使用 camelCase，后端通过 Jackson 统一转换。
2. 小程序 WebSocket 固定使用 `/ws/v1/client` 单连接，客户端连接后订阅多个设备。禁止改成一个设备一个连接。
3. MQTT topic 必须保持本文和既有文档定义，不得改名。
4. 不得明文保存或返回 `openid`、`device_secret`、`wifi_password`、JWT、MinIO `secret_key`。
5. 所有设备和事件接口必须校验当前用户是否已绑定目标设备。
6. Controller 只做参数接收和返回，不写复杂业务逻辑。业务逻辑放在 Service、Handler 或专用组件中。

## 2. 系统边界

### 2.1 模块说明

| 模块 | 目录 | 作用 |
| --- | --- | --- |
| 后端服务 | `campus-server` | HTTP API、JWT 鉴权、MQTT 接入、WebSocket 推送、MinIO URL 生成、数据库持久化 |
| 微信小程序 | `campus-miniprogram` | 用户登录、设备绑定、事件查看、实时事件和设备状态接收 |
| 管理后台 | `campus-admin-web` | 管理端配置查看/更新、系统信息查看 |
| SQL | `sql/campus.sql` | 数据库初始化脚本 |
| 文档 | `docs/` | 拆分后的接口、协议、部署和存储说明 |

### 2.2 运行时依赖

| 依赖 | 用途 |
| --- | --- |
| MySQL | 保存用户、设备、绑定关系、事件、WiFi 配置 |
| Redis | MQTT 去重、设备在线 TTL、JWT 黑名单、临时 URL 和会话相关缓存 |
| MinIO | 保存报警音频、头像等对象文件 |
| EMQX/MQTT Broker | 接收设备上报，向设备下发上传 URL 或配置 |
| 微信小程序平台 | 获取 `code` 并换取 openid，openid 只取哈希后保存 |

## 3. 通用 HTTP 约定

### 3.1 Base URL

后端本地默认端口：

```text
http://localhost:8080
```

小程序端 `services/request.js` 会把业务接口拼到 `config/env.js` 的 `baseUrl` 后面。当前小程序服务层使用的相对路径是 `/auth/...`、`/devices/...`，因此 `baseUrl` 通常应配置到 `/api/v1` 层级。

管理后台 Axios 使用：

```text
/backend/v1
```

### 3.2 统一响应结构

所有 HTTP API 返回：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

字段说明：

| 字段 | 类型 | 用途 |
| --- | --- | --- |
| `code` | integer | 业务状态码。`0` 表示成功；`401` 表示未登录或 token 失效；`403` 表示无权限；其他值表示业务失败 |
| `message` | string | 给前端展示或调试的说明 |
| `data` | object/array/null | 具体业务数据 |

### 3.3 分页结构

分页接口返回：

```json
{
  "records": [],
  "total": 0,
  "page": 1,
  "size": 10
}
```

字段说明：

| 字段 | 类型 | 用途 |
| --- | --- | --- |
| `records` | array | 当前页数据 |
| `total` | integer | 满足条件的总记录数 |
| `page` | integer | 当前页码，从 `1` 开始 |
| `size` | integer | 每页数量，后端会限制到配置的最大值 |

### 3.4 鉴权

除登录和刷新 token 外，HTTP API 使用 Bearer Token：

```http
Authorization: Bearer <access_token>
```

开放接口：

| 接口 | 说明 |
| --- | --- |
| `POST /api/v1/auth/wx/login` | 微信小程序登录 |
| `POST /api/v1/auth/openid/admin-login` | 管理后台 openid 登录，仅用于已登记 ADMIN 用户 |
| `POST /api/v1/auth/refresh` | 刷新 access token |
| `/v3/api-docs/**`、`/swagger-ui/**` | 当前实现匿名开放接口文档；生产环境应关闭或加访问控制 |
| `/ws/v1/client` | WebSocket 握手，握手阶段自行解析 token |

管理后台接口 `/backend/v1/**` 要求当前 JWT 角色为 `ADMIN`。

当前代码中 `application.yml` 默认激活 `dev` profile，并包含本地/演示环境连接参数。真实生产部署时必须改为外部化配置，不能把数据库、Redis、MQTT、微信、MinIO、JWT 密钥写入仓库。

### 3.5 时间格式

后端统一使用 Asia/Shanghai 时区。`LocalDateTime` 返回格式：

```text
yyyy-MM-dd HH:mm:ss
```

查询参数中的 `start_time`、`end_time` 也使用同样格式。

## 4. 字段命名规则

后端 Java 类通常使用 camelCase，例如 `deviceId`、`accessToken`。对外 JSON 统一转为 `snake_case`：

| Java 字段 | JSON 字段 |
| --- | --- |
| `accessToken` | `access_token` |
| `refreshToken` | `refresh_token` |
| `userInfo` | `user_info` |
| `deviceId` | `device_id` |
| `eventTime` | `event_time` |
| `onlineStatus` | `online_status` |
| `lastOnlineTime` | `last_online_time` |

二次开发新增 DTO/VO 时，只需遵守 Java 命名规范，确认 Jackson 的 `SNAKE_CASE` 配置没有被覆盖即可。

## 5. 状态枚举

### 5.1 用户角色 `role`

| 值 | 用途 |
| --- | --- |
| `USER` | 普通小程序用户 |
| `ADMIN` | 管理后台用户，可访问 `/backend/v1/**` |

### 5.2 设备在线状态 `online_status`

| 值 | 用途 |
| --- | --- |
| `ONLINE` | Redis 中存在 `device:online:{device_id}`，设备心跳未过期 |
| `OFFLINE` | Redis 在线 key 不存在或设备主动上报离线 |

### 5.3 事件文件状态 `file_status`

| 值 | 用途 |
| --- | --- |
| `UPLOADING` | 后端已收到报警并下发 MinIO 上传 URL，等待设备上传确认 |
| `SUCCESS` | 设备确认文件上传成功 |
| `FAILED` | 设备确认文件上传失败 |

### 5.4 事件推送状态 `push_status`

| 值 | 用途 |
| --- | --- |
| `PENDING` | 尚未成功推给在线客户端，需要后续补拉 |
| `PUSHED` | 已通过 WebSocket 推送或已被 `/events/unpulled` 补拉 |
| `FAILED` | 预留状态，表示推送失败 |

### 5.5 阅读状态 `read_status`

| 值 | 用途 |
| --- | --- |
| `UNREAD` | 用户未确认阅读 |
| `READ` | 用户已确认阅读 |

## 7. HTTP API 详细说明

### 7.1 微信登录

```http
POST /api/v1/auth/wx/login
Content-Type: application/json
```

请求：

```json
{
  "code": "wx.login 返回的临时 code"
}
```

请求字段：

| 字段 | 类型 | 必填 | 用途 |
| --- | --- | --- | --- |
| `code` | string | 是 | 小程序 `wx.login` 获取的临时登录凭证，后端用它向微信换取 openid |

响应 `data`：

```json
{
  "access_token": "jwt access token",
  "refresh_token": "jwt refresh token",
  "token_type": "Bearer",
  "expires_in": 7200,
  "user_info": {
    "user_id": "usr_xxx",
    "nickname": "微信用户",
    "avatar_url": null,
    "phone": null,
    "email": null,
    "role": "USER",
    "created_at": "2026-05-24 12:00:00",
    "is_new_user": true
  }
}
```

响应字段：

| 字段 | 类型 | 用途 |
| --- | --- | --- |
| `access_token` | string | 短期访问 token。只返回给客户端，不写入日志，不在接口文档中填真实值 |
| `refresh_token` | string | 刷新 token。只返回给客户端，不写入日志 |
| `token_type` | string | 固定为 `Bearer` |
| `expires_in` | integer | access token 有效期，单位秒 |
| `user_info` | object | 当前用户资料 |
| `user_info.is_new_user` | boolean | 是否首次登录。小程序用它决定是否弹出头像/昵称补全弹窗 |

业务规则：

1. 后端拿 `code` 换取 openid。
2. 只保存 `openid_hash`，不保存明文 openid。
3. 如果用户不存在，则创建新用户，默认昵称为 `微信用户`，角色为 `USER`。
4. 如果用户已存在，直接签发 token，不要求重新填写头像和昵称。

```http
POST /api/v1/auth/openid/admin-login
Content-Type: application/json
```

请求：

```json
{
  "openid": "管理员 openid"
```

请求字段：

| --- | --- | --- | --- |
| `openid` | string | 是 | 后端对该值做 SHA-256 后匹配 `user.openid_hash`，且用户角色必须为 `ADMIN` |

响应 `data` 与微信登录相同，包含 `access_token`、`refresh_token`、`token_type`、`expires_in` 和 `user_info`。

当前实现说明：

1. 该接口在 `SecurityConfig` 中匿名开放，成功登录后才能访问 `/backend/v1/**`。
2. 服务端不保存明文 openid，但请求体会经过网络传输；生产环境应改成更正式的管理员登录方案，至少增加密码、验证码、企业微信/统一身份认证或后台白名单。
3. 管理后台前端当前将 `admin_access_token` 和 `admin_refresh_token` 保存在 `localStorage`。

### 7.3 刷新 access token

```http
POST /api/v1/auth/refresh
Content-Type: application/json
```

请求：

```json
{
  "refresh_token": "refresh token"
}
```

响应 `data`：

```json
{
  "access_token": "new access token",
  "token_type": "Bearer",
  "expires_in": 7200
}
```

字段说明：

| 字段 | 类型 | 用途 |
| --- | --- | --- |
| `refresh_token` | string | 登录接口签发的刷新 token |
| `access_token` | string | 新 access token |
| `expires_in` | integer | 新 access token 有效期 |

小程序 `services/request.js` 在收到 `401` 或 `body.code === 401` 时会自动调用该接口刷新 token 并重试一次原请求。

### 7.4 退出登录

```http
POST /api/v1/auth/wx/logout
Authorization: Bearer <access_token>
```

响应 `data` 为 `null`。

业务规则：

1. 后端读取当前 access token 的 token id。
2. 写入 Redis `jwt:blacklist:{token_id}`。
3. 黑名单 TTL 使用配置 `campus.cache.jwt-blacklist-ttl-seconds`。
4. 客户端本地清除 access token 和 refresh token。

| 字段 | 类型 | 必填 | 限制 | 用途 |

### 7.7 上传头像

```http
POST /api/v1/users/me/avatar
Authorization: Bearer <access_token>
Content-Type: multipart/form-data
```

表单字段：

| 字段 | 类型 | 必填 | 用途 |
| --- | --- | --- | --- |
| `file` | file | 是 | 头像文件 |

响应 `data`：

```json
{
  "avatar_url": "https://example.com/avatar/usr_xxx.jpg"
}
```

规则：

1. 文件不能为空。
2. 大小不能超过 `campus.storage.avatar-max-size-mb`。
3. 扩展名只支持 `jpg`、`jpeg`、`png`、`webp`。
4. 上传成功后更新用户表 `avatar_url`。

| 字段 | 类型 | 必填 | 限制 | 用途 |
| --- | --- | --- | --- | --- |
| `device_id` | string | 是 | 最长 64 | 要绑定的设备 ID |
| `bind_code` | string | 是 | 最长 16 | Redis 中 `device:bind_code:{device_id}` 保存的一次性绑定码 |

## 9. MQTT 协议

### 9.1 Topic 约定

设备发布到后端：

| Topic | 方向 | 用途 |
| --- | --- | --- |
| `device/{product_type}/{device_id}/alarm/post` | 设备 -> 后端 | 上报警情，申请上传 URL |
| `device/{product_type}/{device_id}/alarm/confirm` | 设备 -> 后端 | 告知报警文件上传结果 |
| `device/{product_type}/{device_id}/bind` | 设备 -> 后端 | 上报一次性绑定码 |
| `device/{product_type}/{device_id}/status/online` | 设备 -> 后端 | 上报在线或离线状态 |
| `device/{product_type}/{device_id}/config/wifi/set/reply` | 设备 -> 后端 | 回复 WiFi 设置结果 |
| `device/{product_type}/{device_id}/config/wifi/list` | 设备 -> 后端 | 上报设备保存的 WiFi 配置列表 |

后端发布给设备：

| Topic | 方向 | 用途 |
| --- | --- | --- |
| `device/{product_type}/{device_id}/alarm/upload` | 后端 -> 设备 | 下发 MinIO 上传 URL |
| `device/{product_type}/{device_id}/config/wifi/set` | 后端 -> 设备 | 下发 WiFi 配置 |

重要：topic 中的 `{product_type}`、`{device_id}` 必须和 payload 中同名字段一致，否则后端忽略该消息。

### 9.2 MQTT 消息通用字段

| 字段 | 类型 | 必填 | 用途 |
| --- | --- | --- | --- |
| `mqtt_msg_id` | string | 是 | 消息唯一 ID，用于 Redis 和数据库去重 |
| `product_type` | string | 建议必填 | 产品类型，必须和 topic 一致 |
| `device_id` | string | 建议必填 | 设备 ID，必须和 topic 一致 |

`mqtt_msg_id` 推荐格式：

```text
13位毫秒时间戳-3位自增序号-device_id
```

去重策略：

1. Redis key：`mqtt:dedup:{device_id}:{mqtt_msg_id}`。
2. 数据库唯一索引兜底：`event(device_table_id, mqtt_msg_id)`。


```json

  "product_type": "anti_bullying",
  "device_id": "dev001",
```
```json
  "event_id": "evt_xxx",
  "object_key": "anti_bullying/audio/dev001/2026/05/24/evt_xxx.wav",
```

| 字段 | 类型 | 用途 |

| --- | --- | --- |
| `max_size` | integer | 设备最多保存的 WiFi 配置数量 |
| `id_using` | integer | 当前正在使用的配置编号 |
| `config` | array | WiFi 配置列表 |
| `config[].config_id` | integer | 设备内部配置编号 |
| `config[].wifi_name` | string | WiFi 名称 |

后端处理：按 `(device_table_id, config_id)` 插入或更新 `device_config_wifi`。不保存 WiFi 密码。

### 9.9 WiFi 设置回复 `config/wifi/set/reply`

Topic：

```text
device/{product_type}/{device_id}/config/wifi/set/reply
```

Payload：

```json
{
  "mqtt_msg_id": "1716537600000-001-dev001",
  "config_id": 1,
  "success": true
}
```

字段说明：

| 字段 | 类型 | 用途 |
| --- | --- | --- |
| `config_id` | integer | 被设置的配置编号 |
| `success` | boolean | 设备是否设置成功 |

当前实现只记录日志，后续可扩展为写入配置任务表或通知管理端。

### 9.10 后端下发 WiFi 设置 `config/wifi/set`

Topic：

```text
device/{product_type}/{device_id}/config/wifi/set
```

建议 Payload：

```json
{
  "mqtt_msg_id": "server-1716537600000-dev001",
  "config_id": 1,
  "wifi_name": "Campus-WiFi",
  "wifi_password": "仅通过 MQTT 临时下发，不落库"
}
```

注意：当前项目已有发布器方法 `publishWifiSet`，但没有暴露 HTTP 接口。二开时如果新增后台设置 WiFi 功能：

1. 不得将 `wifi_password` 明文保存到数据库。
2. 日志中不得打印 `wifi_password`。
3. MQTT topic 必须使用 `device/{product_type}/{device_id}/config/wifi/set`。

## 10. MinIO 文件存储

### 10.1 报警文件对象 Key

默认模板：

```text
{product_type}/audio/{device_id}/{yyyy}/{MM}/{dd}/{event_id}.{ext}
```

字段说明：

| 占位符 | 来源 | 用途 |
| --- | --- | --- |
| `{product_type}` | `device.product_type` | 按产品类型分目录 |
| `{device_id}` | `device.device_id` | 按设备分目录 |
| `{yyyy}`、`{MM}`、`{dd}` | 当前日期 | 按日期分目录 |
| `{event_id}` | 后端生成 | 文件和事件关联 |
| `{ext}` | `file_name` 扩展名 | 文件类型 |

### 10.2 头像对象 Key

默认模板：

```text
avatar/{user_id}.{ext}
```

### 10.3 文件 URL 原则

1. 数据库长期保存 `file_key`，不长期保存访问 URL。
2. 播放或下载报警文件前，调用 `/events/{event_id}/refresh-url` 获取临时 URL。
3. 头像当前返回 `avatar_url`，用于页面展示。
4. 不在日志中打印完整预签名 URL。

## 11. Redis Key 说明

### 11.1 缓存配置字段与功能对应表

| 配置字段 | Redis Key / 相关逻辑 | 影响功能 | 默认值 | 修改影响 |
| --- | --- | --- | --- | --- |
| `campus.cache.mqtt_dedup_ttl_seconds` | `mqtt:dedup:{device_id}:{mqtt_msg_id}` | MQTT 消息去重，防止设备重复上报导致重复建事件 | 86400 秒 | 调大可延长重复消息识别窗口；调小会更早允许相同 `mqtt_msg_id` 再次进入处理，仍有数据库唯一索引兜底 |
| `campus.cache.device_online_ttl_seconds` | `device:online:{device_id}`、`device:last_heartbeat:{device_id}` | 设备在线状态判断、设备列表 `online_status`、WebSocket `DEVICE_STATUS` | 90 秒 | 调大后设备离线显示会更慢；调小后心跳稍慢就可能显示离线 |
| `campus.cache.bind_code_ttl_seconds` | `device:bind_code:{device_id}` | 小程序绑定设备时的一次性绑定码有效期 | 60 秒 | 调大方便用户输入；调小更安全但更容易过期 |
| `campus.cache.access_url_ttl_seconds` | MinIO 访问 URL 默认过期时间，预留 `minio:access_url:{event_id}` | 事件音频/文件播放前调用 `refresh-url` 获取的临时访问 URL | 3600 秒 | 调大减少刷新 URL 次数；调小降低 URL 泄露后的可用时间 |
| `campus.cache.upload_url_ttl_seconds` | MQTT `alarm/upload` 下发的 `upload_url` 有效期 | 设备报警文件直传 MinIO 的上传窗口 | 300 秒 | 调大适合弱网或大文件；调小会增加设备上传超时概率 |
| `campus.cache.jwt_blacklist_ttl_seconds` | `jwt:blacklist:{token_id}` | 退出登录后 access token 拉黑 | 7200 秒 | 通常应不小于 access token 有效期，否则退出后的 token 可能提前恢复可用 |
| `campus.cache.unpulled_event_cache_ttl_seconds` | 预留 `event:unpulled:user:{user_id}` | 未推送事件补拉缓存扩展位 | 300 秒 | 当前核心逻辑直接查数据库；二开缓存未拉取事件时使用 |

这些字段主要在 `application.yml` 的 `campus.cache` 下配置，管理后台 `/backend/v1/config` 当前可查看和运行时更新其中一部分：`mqtt_dedup_ttl_seconds`、`device_online_ttl_seconds`、`bind_code_ttl_seconds`、`access_url_ttl_seconds`、`upload_url_expire_seconds`、`access_url_expire_seconds`。运行时更新不会写回配置文件，服务重启后仍以配置文件为准。

### 11.2 Redis Key 明细

| Key | 用途 | TTL |
| --- | --- | --- |
| `mqtt:dedup:{device_id}:{mqtt_msg_id}` | MQTT 第一层去重 | `mqtt_dedup_ttl_seconds` |
| `device:online:{device_id}` | 判断设备在线 | `device_online_ttl_seconds` |
| `device:last_heartbeat:{device_id}` | 记录最后一次心跳时间戳 | `device_online_ttl_seconds` |
| `device:bind_code:{device_id}` | 一次性绑定码 | `bind_code_ttl_seconds` |
| `jwt:blacklist:{token_id}` | 已退出登录或失效 token | `jwt_blacklist_ttl_seconds` |
| `event:unpulled:user:{user_id}` | 预留的未拉取事件缓存 | `unpulled_event_cache_ttl_seconds` |
| `minio:access_url:{event_id}` | 预留的访问 URL 缓存 | `access_url_ttl_seconds` |
| `ws:user:{user_id}` | 预留的 WebSocket 用户会话索引 | 视实现而定 |
| `ws:device:{device_id}` | 预留的 WebSocket 设备订阅索引 | 视实现而定 |

当前 WebSocket 会话索引主要保存在 JVM 内存的 `WebSocketSessionManager` 中。多实例部署时需要扩展为 Redis 或消息总线，否则不同实例之间无法互推订阅消息。

## 12. 前端调用流程

### 12.1 小程序登录流程

1. 用户点击“微信一键登录”。
2. 小程序调用 `wx.login` 获取 `code`。
3. 调用 `POST /auth/wx/login`。
4. 保存 `access_token` 和 `refresh_token`。
5. 如果 `user_info.is_new_user = true`，弹出头像和微信名称填写。
6. 老用户直接 `switchTab` 到主页。

### 12.2 小程序设备和事件流程

1. 进入主页或设备页，调用设备列表接口。
2. 使用绑定设备 ID 列表建立 WebSocket 订阅。
3. 收到 `NEW_EVENT` 后将事件插到列表顶部。
4. 收到 `DEVICE_STATUS` 后更新对应设备在线状态。
5. WebSocket 重连后调用 `/events/unpulled` 做补偿。

### 12.3 小程序音频播放流程

1. 用户进入事件详情。
2. 点击播放。
3. 调用 `POST /events/{event_id}/refresh-url`。
4. 使用返回的 `file_url` 创建 `InnerAudioContext` 播放。

### 12.4 管理后台流程

1. 管理后台从 `localStorage.admin_access_token` 读取 token。
2. Axios 请求 `/backend/v1/**`。
3. 后端通过 `@PreAuthorize("hasRole('ADMIN')")` 和 SecurityConfig 双重限制管理接口。

## 13. 权限和安全检查清单

新增或修改接口时逐项检查：

| 检查项 | 要求 |
| --- | --- |
| JSON 字段 | 对外保持 `snake_case` |
| JWT | 不在响应、日志、文档样例中写真实 token |
| openid | 只保存 `openid_hash` |
| 设备密钥 | 只保存哈希，不返回 `device_secret` |
| WiFi 密码 | 不落库，不打印日志 |
| MinIO 密钥 | 管理端只返回脱敏值 |
| 设备接口 | 必须校验当前用户已绑定设备 |
| 事件接口 | 必须通过事件所属设备校验绑定关系 |
| WebSocket | 只使用 `/ws/v1/client`，订阅时校验设备绑定 |
| MQTT topic | 不改名，不随意新增不兼容 topic |
| Controller | 不写复杂业务逻辑 |

## 14. 二次开发建议

### 14.1 新增小程序接口

建议流程：

1. 新增 DTO，字段使用 Java camelCase。
2. Controller 只接收参数并调用 Service。
3. Service 中通过 `SecurityContextUtil.currentUser()` 获取当前用户。
4. 如果涉及设备，先调用 `DeviceService.ensureUserBoundDevice(...)`。
5. 返回 VO，不返回 Entity。
6. 补充本文档的接口说明和字段说明。

### 14.2 新增设备协议

建议流程：

1. 保持 topic 前缀 `device/{product_type}/{device_id}/...`。
2. Payload 必须有 `mqtt_msg_id`。
3. 在 `MqttReceiver` 中订阅新 action。
4. 在 `MqttMessageHandler` 实现类中处理业务。
5. Handler 中避免直接散落复杂存储逻辑，可拆 Service。
6. 必须考虑 Redis 去重和数据库唯一约束兜底。

### 14.3 多实例部署

当前项目可单实例运行。若部署多个后端实例，需要重点处理：

1. WebSocket 会话在 JVM 内存，跨实例无法直接推送。
2. MQTT 消息可能只被某个实例消费，需要统一推送到正确的 WebSocket 实例。
3. 可选方案：Redis Pub/Sub、消息队列、或集中式 WebSocket Gateway。
4. JWT 黑名单和设备在线状态已经在 Redis 中，天然支持多实例共享。

### 14.4 查询参数 `snake_case` 兼容

请求体 JSON 已由 Jackson 处理 `snake_case`。但 GET 查询参数的绑定在 Spring MVC 中不一定自动把 `read_status` 绑定到 `readStatus`。为了避免前后端不一致，建议对 GET 查询 DTO 或 Controller 参数显式处理：

```java
@RequestParam(name = "read_status", required = false) String readStatus
```

或增加统一参数转换策略。当前小程序已按 `snake_case` 发送查询参数。

## 15. 常见联调问题

### 15.1 小程序登录后老用户仍要求填头像昵称

检查登录接口返回的：

```json
{
  "user_info": {
    "is_new_user": false
  }
}
```

老用户应直接进入主页。只有 `is_new_user=true` 才应弹出资料补全。

### 15.2 设备一直离线

检查：

1. 设备是否发布 `device/{product_type}/{device_id}/status/online`。
2. Payload 中 `device_id` 是否和 topic 一致。
3. Redis 是否存在 `device:online:{device_id}`。
4. `device_online_ttl_seconds` 是否过短。

### 15.3 收不到新事件推送

检查：

1. 小程序是否连接 `/ws/v1/client`。
2. WebSocket 是否发送 `SUBSCRIBE_EVENTS`。
3. 订阅的 `device_ids` 是否已绑定当前用户。
4. 设备是否发布 `alarm/confirm`。
5. 如果推送失败，调用 `/events/unpulled` 是否能补拉到 `PENDING` 事件。

### 15.4 播放音频失败

检查：

1. 事件 `file_status` 是否为 `SUCCESS`。
2. 事件是否存在 `file_key`。
3. `/events/{event_id}/refresh-url` 是否返回有效 `file_url`。
4. MinIO bucket、endpoint 和预签名 URL 是否可从小程序网络访问。

### 15.5 绑定设备失败

检查：

1. `device` 表是否存在对应 `device_id`。
2. Redis 是否存在 `device:bind_code:{device_id}`。
3. 请求中的 `bind_code` 是否正确且未过期。
4. 当前用户是否已登录，Authorization 是否有效。
