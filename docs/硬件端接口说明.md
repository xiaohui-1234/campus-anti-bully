# 硬件端接口说明

本文面向 STM32 主控、ESP8266、TF 卡和后端联调。当前硬件代码位于：

```text
stm32主控源码/联网测试win
```

## 1. 当前硬件职责

- 通过 ESP8266 连接 WiFi 和 MQTT Broker。
- 周期发布设备在线状态。
- 按键或语音触发报警后录制 WAV 文件。
- 将报警元数据通过 MQTT `alarm/post` 上报后端。
- 接收后端 MQTT `alarm/upload` 上传指令。
- 使用 ESP8266 透传模式对 MinIO 预签名 URL 执行 HTTP PUT。
- 上传完成后通过 MQTT `alarm/confirm` 回传结果。
- 使用 TF 卡保存本地 WAV 文件。

## 3. 引脚与外设

### ESP8266

当前头文件：`Hardware/esp8266.h`

| 信号 | 定义 |
| --- | --- |
| USART | `USART2` |
| TX | `PA2` |
| RX | `PA3` |
| CH_PD | `PA5`，当前宏为空操作，若硬件无上拉需实现 |
| RST | `PA8`，已实现拉高/拉低和初始化 |

RST 宏：

```c
#define macESP8266_RST_HIGH_LEVEL() GPIO_SetBits(macESP8266_RST_PORT, macESP8266_RST_PIN)
#define macESP8266_RST_LOW_LEVEL()  GPIO_ResetBits(macESP8266_RST_PORT, macESP8266_RST_PIN)
```

`ESP8266_Init()` 会先初始化 RST GPIO，再执行硬复位。

### TF/Micro SD Card Adapter

当前 SD SPI 驱动：`Hardware/sd_spi.c`

| 信号 | 定义 |
| --- | --- |
| SPI | `SPI1` |
| CS | `PA4` |
| SCK | `PA5` |
| MISO | `PA6` |
| MOSI | `PA7` |

Micro SD Card Adapter 通常没有独立 RST 引脚。当前恢复方式为软复位：

```c
FRESULT TF_Card_Reset(void);
```

实现逻辑：

1. `TF_Card_Unmount()`
2. `SD_SPI_Init()`
3. `TF_Card_Mount()`

`SD_SPI_Init()` 内部会重新配置 GPIO/SPI、拉高 CS、发送空时钟并执行 SD 卡初始化命令。

### 通用 GPIO 库

新增于：

```text
System/bsp_gpio.h
System/bsp_gpio.c
```

提供：

- 自动开 GPIO 时钟
- 通用引脚初始化
- 输入/输出初始化
- 置位、复位、写电平、翻转
- 读取输入/输出电平

## 5. MQTT 连接流程

ESP8266 初始化后执行：

1. 配置 USART。
2. 连接 WiFi。
3. 配置 MQTT 用户：

```text
AT+MQTTUSERCFG=0,1,"client_id","username","password",0,0,""
```

4. 连接 Broker：

```text
AT+MQTTCONN=0,"host",1883,0
```

5. 订阅上传指令：

```text
AT+MQTTSUB=0,"device/{product_type}/{device_id}/alarm/upload",1
```

6. 发布在线状态。

当前代码通过 `Campus_MQTT_Service(nowMs)` 在主循环周期发布在线心跳。

## 6. 在线状态

Topic：

```text
device/{product_type}/{device_id}/status/online
```

Payload：

```json
{
  "mqtt_msg_id": "1779177600000-001-dev001",
  "product_type": "anti_bullying",
  "device_id": "dev001",
  "status": "online",
  "timestamp": 1779177600000
}
```

后端收到后会刷新 Redis 在线 TTL，更新 `last_online_time`，并通过 WebSocket 推送 `DEVICE_STATUS`。

## 7. 报警上报

当按键或语音触发报警：

1. `Alarm_StartRecord(eventType, alarmInfo)` 保存待处理报警上下文。
2. 主控开始录音并保存 WAV 到 TF 卡。
3. 录音完成后调用 `Campus_MQTT_PostAlarmForFile(wavPath, fileSize)`。
4. 设备发布 `alarm/post`。

Topic：

```text
device/{product_type}/{device_id}/alarm/post
```

Payload：

```json
{
  "mqtt_msg_id": "1779177600123-001-dev001",
  "product_type": "anti_bullying",
  "device_id": "dev001",
  "event_type": "VOICE",
  "alarm_info": "fight",
  "file_name": "alarm.wav",
  "file_size": 123456,
  "content_type": "audio/wav",
  "timestamp": 1779177600123
}
```

后端生成事件和预签名上传 URL。

## 8. 接收上传指令

设备订阅：

```text
device/{product_type}/{device_id}/alarm/upload
```

Payload：

```json
{
  "mqtt_msg_id": "1779177600123-001-dev001",
  "event_id": "evt_dev001_1779177600123",
  "object_key": "anti_bullying/audio/dev001/2026/06/08/evt_dev001_1779177600123.wav",
  "upload_url": "http://oss.example.com/campus/...",
  "expire_seconds": 300
}
```

当前 STM32 代码要求：

- 能从 ESP8266 `+MQTTSUBRECV` 回包中提取 payload。
- `upload_url` 必须是设备可访问的 HTTP URL。
- 代码会检查预签名 URL 中的 `X-Amz-Algorithm`、`X-Amz-Credential`、`X-Amz-SignedHeaders`、`X-Amz-Signature`。
- 使用 `mqtt_msg_id + event_id` 做本地环形缓存去重。

## 9. HTTP PUT 上传

上传流程：

1. 解析 `upload_url` 的 host、port、path。
2. ESP8266 建立 TCP 连接。
3. 进入透传模式。
4. 发送 HTTP PUT Header。
5. 从 TF 卡分块读取 WAV 文件并发送。
6. 等待 `HTTP/1.1 200` 或 `HTTP/1.1 201`。

当前限制：

- `CAMPUS_HTTP_UPLOAD_CHUNK_SIZE` 默认 `256`。
- `CAMPUS_HTTP_UPLOAD_RESPONSE_TIMEOUT_MS` 默认 `90000`。
- 代码当前只支持普通 HTTP 上传；如果 MinIO 预签名 URL 是 HTTPS，ESP8266 侧需要额外适配 TLS 或改用可访问的 HTTP 公网端点。

## 10. 上传确认

Topic：

```text
device/{product_type}/{device_id}/alarm/confirm
```

Payload：

```json
{
  "mqtt_msg_id": "1779177600456-002-dev001",
  "product_type": "anti_bullying",
  "device_id": "dev001",
  "event_id": "evt_dev001_1779177600123",
  "object_key": "anti_bullying/audio/dev001/2026/06/08/evt_dev001_1779177600123.wav",
  "upload_status": "success"
}
```

`upload_status` 为 `success` 时后端标记 `file_status=SUCCESS`，否则标记 `FAILED`。

如果收到重复 `alarm/upload`，设备会重新发送之前缓存的确认结果，避免重复上传。

## 11. 绑定码

当前硬件文档要求设备支持发布：

```text
device/{product_type}/{device_id}/bind
```

Payload：

```json
{
  "mqtt_msg_id": "1779177600000-001-dev001",
  "product_type": "anti_bullying",
  "device_id": "dev001",
  "bind_code": "123456"
}
```

后端将绑定码写入 Redis，用户在小程序端提交绑定码完成绑定。

## 12. WiFi 配置列表

设备可发布：

```text
device/{product_type}/{device_id}/config/wifi/list
```

Payload：

```json
{
  "mqtt_msg_id": "1779177600000-001-dev001",
  "max_size": 5,
  "id_using": 1,
  "config": [
    {
      "config_id": 1,
      "wifi_name": "Campus-WiFi"
    }
  ]
}
```

后端只保存 `wifi_name`，不得上传或保存 `wifi_password`。

## 13. 本地文件系统

TF 卡模块：

```c
FRESULT TF_Card_Mount(void);
FRESULT TF_Card_Unmount(void);
FRESULT TF_Card_Reset(void);
FRESULT TF_Card_AppendFile(const char *path, const void *data, UINT len, UINT *written);
```

SPI 底层：

```c
SD_SPI_Status SD_SPI_Init(void);
SD_SPI_Status SD_SPI_ReadBlocks(uint32_t lba, uint8_t *buf, uint32_t count);
SD_SPI_Status SD_SPI_WriteBlocks(uint32_t lba, const uint8_t *buf, uint32_t count);
uint32_t SD_SPI_GetSectorCount(void);
```

原来 SPI 初始化由 `TF_Card_Mount()` 触发 FatFs `disk_initialize()`，最终调用 `SD_SPI_Init()`。`TF_Card_Reset()` 中显式调用 `SD_SPI_Init()` 是为了强制重新初始化 SPI 和 SD 卡通信流程。

## 14. 联调检查清单

- WiFi SSID/密码正确，且不要提交真实值。
- EMQX host/port 对 ESP8266 可达。
- 后端 `campus.mqtt.enabled=true`。
- 设备 `product_type`、`device_id` 与数据库/后端一致。
- topic 与文档一致。
- MQTT payload 字段使用 `snake_case`。
- `mqtt_msg_id` 每次上报唯一。
- TF 卡可以挂载、录音文件可以读取。
- MinIO 上传 URL 对设备是 HTTP 可访问地址。
- 上传确认中的 `object_key` 必须与后端下发一致。
- 小程序用户已绑定设备，否则事件和 WebSocket 推送不会对该用户可见。
