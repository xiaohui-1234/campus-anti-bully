# 硬件端接口说明

最后更新：2026-05-25

本文说明当前软件系统面向硬件设备开放和约定的接口。硬件设备主要通过 MQTT 与后端通信，报警文件通过后端下发的 MinIO 预签名 URL 上传。

本文只描述当前后端已经实现的软件侧协议，不包含 STM32 固件实现细节。

## 1. 接入范围

当前软件侧已实现以下硬件端能力：

| 能力 | 状态 | 说明 |
| --- | --- | --- |
| 设备在线状态上报 | 已实现 | 设备通过 MQTT 上报在线/离线，后端更新 Redis 和数据库 |
| 设备绑定码上报 | 已实现 | 设备通过 MQTT 上报一次性绑定码，小程序用该绑定码绑定设备 |
| 报警事件上报 | 已实现 | 设备上报警情，后端创建事件并下发上传 URL |
| 报警文件上传确认 | 已实现 | 设备上传文件后通过 MQTT 确认成功/失败 |
| WiFi 列表上报 | 已实现后端接收 | 后端可保存设备上报的 WiFi 配置名，不保存密码 |
| WiFi 设置结果回复 | 已实现后端接收 | 当前仅记录日志 |
| WiFi 设置下发 | 有发布器，无 HTTP 入口 | 后端有 MQTT 发布方法，但当前没有管理端接口触发 |

## 2. 设备身份

硬件设备在软件系统中的身份由 `device` 表维护。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `device_id` | string | 设备对外唯一标识，也是 MQTT topic 中的设备标识 |
| `product_type` | string | 产品类型，也是 MQTT topic 中的产品分组 |
| `device_secret` | string | 设备密钥哈希值，当前后端不校验 MQTT 设备签名 |

设备发送 MQTT 前，后端数据库必须已有对应设备记录。否则后端会返回业务异常并忽略该设备消息。

示例设备：

```text
product_type = anti_bullying
device_id = dev001
```

## 3. MQTT 连接

MQTT Broker 由后端配置决定：

```yaml
campus:
  mqtt:
    broker-url: tcp://47.116.107.124:1883
    username: root
    password: 配置值
    qos: 1
```

当前后端使用 Eclipse Paho 连接 Broker，并订阅以下通配 topic：

```text
device/+/+/alarm/post
device/+/+/alarm/confirm
device/+/+/bind
device/+/+/status/online
device/+/+/config/wifi/set/reply
device/+/+/config/wifi/list
```

硬件端应使用与 Broker 匹配的账号密码接入。当前后端不负责 MQTT Broker 侧 ACL，下列权限应在 Broker 配置中保证：

| 方向 | Topic |
| --- | --- |
| 设备发布 | `device/{product_type}/{device_id}/alarm/post` |
| 设备发布 | `device/{product_type}/{device_id}/alarm/confirm` |
| 设备发布 | `device/{product_type}/{device_id}/bind` |
| 设备发布 | `device/{product_type}/{device_id}/status/online` |
| 设备发布 | `device/{product_type}/{device_id}/config/wifi/set/reply` |
| 设备发布 | `device/{product_type}/{device_id}/config/wifi/list` |
| 设备订阅 | `device/{product_type}/{device_id}/alarm/upload` |
| 设备订阅 | `device/{product_type}/{device_id}/config/wifi/set` |

## 4. Topic 格式

统一格式：

```text
device/{product_type}/{device_id}/{action}
```

示例：

```text
device/anti_bullying/dev001/alarm/post
```

后端会从 topic 中解析：

| 位置 | 字段 | 示例 |
| --- | --- | --- |
| 第 1 段 | 固定 `device` | `device` |
| 第 2 段 | `product_type` | `anti_bullying` |
| 第 3 段 | `device_id` | `dev001` |
| 第 4 段及之后 | `action` | `alarm/post` |

如果 payload 中包含 `product_type` 或 `device_id`，后端会要求它们和 topic 一致。不一致时消息会被忽略。

## 5. 通用消息字段

硬件端 payload 使用 JSON，字段为 `snake_case`。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `mqtt_msg_id` | string | 是 | MQTT 业务消息唯一 ID，用于 Redis 去重 |
| `product_type` | string | 建议 | 产品类型，应与 topic 一致 |
| `device_id` | string | 建议 | 设备 ID，应与 topic 一致 |

`mqtt_msg_id` 缺失时，后端会忽略消息。

推荐格式：

```text
{timestamp_ms}-{seq}-{device_id}
```

示例：

```text
1779177600000-001-dev001
```

后端去重规则：

| 层级 | 规则 |
| --- | --- |
| Redis | `mqtt:dedup:{device_id}:{mqtt_msg_id}` |
| 数据库 | `event(device_table_id, mqtt_msg_id)` 唯一索引兜底 |

## 6. 设备在线状态

### 6.1 Topic

```text
device/{product_type}/{device_id}/status/online
```

### 6.2 Payload

```json
{
  "mqtt_msg_id": "1779177600000-001-dev001",
  "product_type": "anti_bullying",
  "device_id": "dev001",
  "status": "online",
  "timestamp": 1779177600000
}
```

### 6.3 字段说明

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `status` | string | 是 | `online` 表示在线；其他值按离线处理 |
| `timestamp` | integer | 否 | 毫秒时间戳；为空时后端使用当前时间 |

### 6.4 后端行为

当 `status=online`：

1. 校验设备存在。
2. 写入 Redis `device:online:{device_id}`。
3. 写入 Redis `device:last_heartbeat:{device_id}`。
4. 更新数据库 `device.last_online_time`。
5. 通过 WebSocket 向订阅该设备的客户端推送 `DEVICE_STATUS`。

当 `status` 不是 `online`：

1. 删除 Redis `device:online:{device_id}`。
2. 通过 WebSocket 推送离线状态。

## 7. 绑定码上报

### 7.1 Topic

```text
device/{product_type}/{device_id}/bind
```

### 7.2 Payload

```json
{
  "mqtt_msg_id": "1779177600000-002-dev001",
  "product_type": "anti_bullying",
  "device_id": "dev001",
  "bind_code": "123456"
}
```

### 7.3 字段说明

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `bind_code` | string | 是 | 小程序绑定设备时输入的一次性绑定码 |

### 7.4 后端行为

1. 校验设备存在。
2. 校验 `bind_code` 非空。
3. 写入 Redis `device:bind_code:{device_id}`。
4. TTL 使用 `campus.cache.bind-code-ttl-seconds`。

小程序绑定接口：

```http
POST /api/v1/devices/bind
```

请求中的 `device_id` 和 `bind_code` 必须与设备发布到 Redis 的值一致。

## 8. 报警事件上报

### 8.1 Topic

```text
device/{product_type}/{device_id}/alarm/post
```

### 8.2 Payload

```json
{
  "mqtt_msg_id": "1779177600000-003-dev001",
  "product_type": "anti_bullying",
  "device_id": "dev001",
  "event_type": "VOICE",
  "alarm_info": "救命",
  "file_name": "alarm.wav",
  "file_size": 240000,
  "content_type": "audio/wav",
  "timestamp": 1779177600000
}
```

### 8.3 字段说明

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `event_type` | string | 是 | 事件类型，如 `BUTTON`、`VOICE`、`FIGHT`、`HELP` |
| `alarm_info` | string | 否 | 报警描述文本 |
| `file_name` | string | 建议 | 后端根据扩展名生成对象 key |
| `file_size` | integer | 否 | 文件大小，当前后端仅接收字段，不做强校验 |
| `content_type` | string | 建议 | 文件 MIME 类型，如 `audio/wav` |
| `timestamp` | integer | 否 | 设备事件发生时间，毫秒时间戳 |

### 8.4 后端行为

1. 校验设备存在。
2. 生成 `event_id`。
3. 根据 `file_name`、设备信息和存储模板生成 `object_key`。
4. 生成 MinIO 预签名上传 URL。
5. 插入事件记录：
   - `file_status=UPLOADING`
   - `push_status=PENDING`
   - `read_status=UNREAD`
6. 发布 `alarm/upload` 给设备。
7. 尝试通过 WebSocket 向订阅该设备的客户端推送 `NEW_EVENT`。

## 9. 后端下发上传 URL

### 9.1 Topic

```text
device/{product_type}/{device_id}/alarm/upload
```

### 9.2 Payload

```json
{
  "mqtt_msg_id": "1779177600000-003-dev001",
  "event_id": "evt_dev001_xxx",
  "object_key": "anti_bullying/audio/dev001/2026/05/25/evt_dev001_xxx.wav",
  "upload_url": "http://example.com/campus/anti_bullying/audio/dev001/2026/05/25/evt_dev001_xxx.wav?...",
  "expire_seconds": 300
}
```

### 9.3 字段说明

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `mqtt_msg_id` | string | 对应设备 `alarm/post` 的消息 ID |
| `event_id` | string | 后端生成的事件 ID |
| `object_key` | string | MinIO 对象 key |
| `upload_url` | string | 预签名 PUT URL |
| `expire_seconds` | integer | URL 有效期，单位秒 |

### 9.4 硬件端期望行为

1. 设备收到 `alarm/upload` 后保存 `event_id`、`object_key`、`upload_url`。
2. 使用 HTTP `PUT` 将报警文件上传到 `upload_url`。
3. 上传成功或失败后发布 `alarm/confirm`。

注意：`upload_url` 是临时 URL，不应长期保存或打印到公开日志。

## 10. 报警文件上传确认

### 10.1 Topic

```text
device/{product_type}/{device_id}/alarm/confirm
```

### 10.2 Payload

上传成功：

```json
{
  "mqtt_msg_id": "1779177600000-004-dev001",
  "product_type": "anti_bullying",
  "device_id": "dev001",
  "event_id": "evt_dev001_xxx",
  "object_key": "anti_bullying/audio/dev001/2026/05/25/evt_dev001_xxx.wav",
  "upload_status": "success"
}
```

上传失败：

```json
{
  "mqtt_msg_id": "1779177600000-004-dev001",
  "product_type": "anti_bullying",
  "device_id": "dev001",
  "event_id": "evt_dev001_xxx",
  "object_key": "anti_bullying/audio/dev001/2026/05/25/evt_dev001_xxx.wav",
  "upload_status": "failed"
}
```

### 10.3 字段说明

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `event_id` | string | 是 | 后端在 `alarm/upload` 中下发的事件 ID |
| `object_key` | string | 建议 | 后端在 `alarm/upload` 中下发的对象 key |
| `upload_status` | string | 是 | `success` 表示成功，其他值按失败处理 |

### 10.4 后端行为

1. 校验设备存在。
2. 查询 `event_id` 对应事件。
3. 校验事件所属设备和 topic 中设备一致。
4. 如果传入 `object_key`，校验它和事件记录的 `file_key` 一致。
5. `upload_status=success` 时更新 `file_status=SUCCESS`。
6. 其他状态更新 `file_status=FAILED`。
7. 尝试 WebSocket 推送 `NEW_EVENT`。
8. 推送失败时，事件保持 `push_status=PENDING`，小程序后续通过 `/api/v1/events/unpulled` 补拉。

## 11. WiFi 配置列表上报

### 11.1 Topic

```text
device/{product_type}/{device_id}/config/wifi/list
```

### 11.2 Payload

```json
{
  "mqtt_msg_id": "1779177600000-005-dev001",
  "product_type": "anti_bullying",
  "device_id": "dev001",
  "max_size": 5,
  "id_using": 1,
  "config": [
    {
      "config_id": 1,
      "wifi_name": "Campus-WiFi"
    }
  ]
}
```

### 11.3 字段说明

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `max_size` | integer | 否 | 设备最多保存的 WiFi 配置数量 |
| `id_using` | integer | 否 | 当前正在使用的配置编号 |
| `config` | array | 否 | WiFi 配置列表 |
| `config[].config_id` | integer | 是 | 设备内部配置编号 |
| `config[].wifi_name` | string | 是 | WiFi 名称 |

### 11.4 后端行为

1. 校验设备存在。
2. 按 `(device_table_id, config_id)` 插入或更新 `device_config_wifi`。
3. 只保存 `wifi_name` 和 `set_time`。
4. 后端不保存 `wifi_password`。

```

### 12.2 Payload

  "device_id": "dev001",
  "config_id": 1,
  "success": true

```

### 12.3 字段说明

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `config_id` | integer | 是 | 被设置的 WiFi 配置编号 |
| `success` | boolean | 是 | 设备是否设置成功 |

### 12.4 后端行为

## 13. 后端下发 WiFi 设置

### 13.1 Topic

```text
device/{product_type}/{device_id}/config/wifi/set
```

### 13.2 Payload

当前后端有 `MqttPublisher.publishWifiSet(...)` 发布能力，但没有 HTTP Controller 或管理后台入口触发。

建议 payload 结构：

```json
{
  "mqtt_msg_id": "server-1779177600000-dev001",
  "config_id": 1,
  "wifi_name": "Campus-WiFi",
  "wifi_password": "仅 MQTT 临时下发，不落库"
}
```

安全要求：

1. `wifi_password` 只能临时下发。
2. 不得写入数据库。
3. 不得写入日志。
4. 不得通过普通查询接口返回。

## 14. 报警事件状态流转

```text
设备 alarm/post
  -> 后端创建事件：UPLOADING / PENDING / UNREAD
  -> 后端发布 alarm/upload
  -> 设备 PUT 文件到 upload_url
  -> 设备 alarm/confirm
  -> 后端更新 SUCCESS 或 FAILED
  -> WebSocket 推送 NEW_EVENT 或等待小程序补拉
```

字段状态：

| 字段 | 可能值 | 说明 |
| --- | --- | --- |
| `file_status` | `UPLOADING` | 已创建事件，等待设备上传 |
| `file_status` | `SUCCESS` | 设备确认上传成功 |
| `file_status` | `FAILED` | 设备确认上传失败或超时 |
| `push_status` | `PENDING` | 未成功推送，等待补拉 |
| `push_status` | `PUSHED` | 已推送或已被补拉 |
| `read_status` | `UNREAD` | 用户未读 |
| `read_status` | `READ` | 用户已读 |

## 15. 软件侧限制和注意事项

1. JSON 字段必须使用 `snake_case`。
2. MQTT topic 必须保持本文定义，不得改名。
3. topic 中的 `product_type`、`device_id` 必须和 payload 保持一致。
4. `mqtt_msg_id` 必须存在，否则后端会忽略消息。
5. `device_id` 必须已登记到数据库。
6. 后端当前不校验 MQTT payload 签名，设备接入安全主要依赖 MQTT Broker 账号和 ACL。
7. 设备密钥 `device_secret` 只应保存哈希，不应明文保存或返回。
8. WiFi 密码不落库、不返回、不打印日志。
9. MinIO 预签名 URL 是临时凭证，不应公开展示。
10. 多后端实例部署时，WebSocket 推送需要额外处理跨实例订阅分发。

## 16. 联调检查

### 16.1 设备在线不显示

检查：

1. topic 是否为 `device/{product_type}/{device_id}/status/online`。
2. `status` 是否为 `online`。
3. 数据库是否存在该 `device_id`。
4. Redis 是否存在 `device:online:{device_id}`。
5. 心跳间隔是否小于 `device-online-ttl-seconds`。

### 16.2 绑定失败

检查：

1. 设备是否发布 `bind`。
2. Redis 是否存在 `device:bind_code:{device_id}`。
3. 小程序提交的 `device_id` 和 `bind_code` 是否匹配。
4. 绑定码是否已过期。

### 16.3 报警事件未生成

检查：

1. topic 是否为 `device/{product_type}/{device_id}/alarm/post`。
2. payload 是否包含 `mqtt_msg_id`。
3. payload 与 topic 的 `device_id` 是否一致。
4. 后端 MinIO 配置是否可用。
5. 后端日志是否有 `Handle alarm/post failed`。

### 16.4 收不到上传 URL

检查：

1. 设备是否订阅 `device/{product_type}/{device_id}/alarm/upload`。
2. 后端是否成功处理 `alarm/post`。
3. MQTT Broker ACL 是否允许后端发布、设备订阅该 topic。

### 16.5 上传确认被忽略

检查：

1. `event_id` 是否来自后端下发。
2. `object_key` 是否和后端下发完全一致。
3. `alarm/confirm` topic 中的 `device_id` 是否和事件所属设备一致。
